**MySQL** is your source of truth. Your application reads and writes here — orders, categories, regions, whatever your domain is. MySQL already writes every row change to a binary log (binlog) for its own replication purposes. That binlog is the entry point for the entire pipeline. You don't modify your application code at all.

**Debezium** connects to MySQL pretending to be a replica. It reads the binlog using MySQL's native replication protocol, parses each entry, and converts it into a structured JSON event containing the operation type (insert/update/delete), the before and after state of the row, a timestamp, and schema metadata. It then publishes that event to Kafka. It also handles the initial snapshot — when you first start it, it reads the full current state of each table, then switches to tailing the binlog for ongoing changes. If Debezium crashes, it tracks its position via Kafka Connect offsets and resumes exactly where it stopped.

**Kafka** is the durable message bus sitting in the middle. Each table gets its own topic — `cdc.source_db.table_a`, `cdc.source_db.table_b`, `cdc.source_db.table_c`. Kafka's role is decoupling and durability. It decouples MySQL from everything downstream — MySQL doesn't know or care who's consuming these events. It provides durability because events are persisted to disk with configurable retention, so if Flink goes down for maintenance, the events queue up and Flink can replay from where it left off. And it enables fan-out — if later you want a second consumer for search indexing or audit logging, it just subscribes to the same topics. Kafka doesn't transform anything. It just stores and delivers messages reliably.

**Zookeeper** manages Kafka's cluster metadata — broker registration, topic configuration, partition leadership elections. It's the coordination layer Kafka depends on. In newer Kafka versions (3.3+) you can use KRaft mode and drop Zookeeper entirely, but the Confluent images in the docker-compose still use it. You don't interact with it directly.

**PyFlink** is where all the intelligence lives. It consumes the three Kafka topics in parallel, deserializes the CDC events, and performs a stateful three-way streaming join. Concretely, Flink holds the latest version of every B row and every C row in its internal state (managed, checkpointed, RocksDB-backed memory). When an event from table A arrives, Flink looks up `A.b_id` in its B-state and `A.c_id` in its C-state, assembles the full denormalized row, and emits it downstream. When a B event arrives, Flink scans its A-state to find every A row referencing that `b_id`, re-joins each one with the updated B data and the existing C data, and re-emits all affected rows. Same logic for C changes. This is the core of your requirement — propagating a change in any source table through a join and into the destination. Flink also manages watermarks for event-time correctness, checkpoints for fault tolerance (so it can recover without reprocessing from scratch), and backpressure (if ClickHouse slows down, Flink pauses consumption from Kafka rather than dropping data).

**ClickHouse** is the analytical destination. It stores table D — the fully denormalized join of A, B, and C. The engine is `ReplacingMergeTree` keyed on `a_id` with a `_version` column, which means when Flink sends an updated row for `a_id = 5`, ClickHouse keeps both the old and new versions temporarily, then deduplicates during background merges by keeping only the highest `_version`. The `table_d_live` view uses `FINAL` to force deduplication at query time so reads always return the latest state. ClickHouse is chosen here because it's columnar, compresses aggressively, and handles analytical queries over this denormalized data orders of magnitude faster than querying the normalized MySQL tables with joins.

**The data flow for a single update:** you run `UPDATE table_a SET amount = 300 WHERE id = 5` in MySQL. MySQL writes this to the binlog. Debezium reads it within ~100ms, wraps it in a JSON envelope, and publishes to the `table_a` Kafka topic. Kafka acknowledges and persists it. Flink picks it up within ~100ms, deserializes it, looks up the corresponding B and C rows from state, builds the joined row `{a_id: 5, amount: 300, b_name: "Electronics", c_region: "North America", ...}`, and POSTs it to ClickHouse via HTTP. ClickHouse inserts it. Total wall time: well under 2 seconds.
